# Production Engineering Instructions

## Mission
Build and operate this API as production software for paying customers. Prioritize safety, observability, tenant isolation, and rollback-friendly changes.

## Stack
- Python 3.11 only
- FastAPI + Uvicorn/Gunicorn
- SQLAlchemy 2.0 async with asyncpg
- PostgreSQL as system of record
- Azure Web App deployment target
- Azure Blob Storage for API call logs

## Architecture Rules
- Single modular FastAPI app under `app/`.
- Domain modules under `app/api/routes` + `app/services`.
- No raw SQL in routers. Use service/repository patterns.
- Keep routes thin; business logic in services.

## Security and Auth
- Support multiple auth providers: local, OIDC/Azure AD, SAML, LDAP, OAuth2 social, API keys.
- Developer auth can exist but must be disabled in production.
- Every non-public route must enforce authentication and authorization.
- Use RBAC checks for route and data access.
- Enforce tenant scope in every query path.

## Data Governance
- Treat tenant isolation as mandatory.
- Prefer DB-level RLS with app-level enforcement as defense in depth.
- Never expose sensitive fields unless explicitly permitted.

## Logging and Audit
- Structured JSON logging only.
- Log request metadata for every API call.
- Record: who accessed what API, tenant, action, status, latency, and data class.
- Persist API audit records to Azure Blob Storage in batch/async mode.
- Never log secrets, passwords, tokens, or MFA values.

## Engineering Quality
- Type hints required on public functions.
- Validate all inputs with Pydantic.
- Use async I/O in request path.
- Add health probes: live and ready.
- Add tests for auth required, RBAC denied, tenant isolation, and happy paths.

## Deployment
- Azure Web App Linux with Python 3.11.
- Use managed identity for Azure resources.
- Secrets sourced from Key Vault or secure app settings.
- Release via staged deployment with smoke checks before production swap.
